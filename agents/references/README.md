# References

Durable background guidance lives here.

References are not step-by-step workflows and are not copyable templates. They capture techniques, conventions, and operational notes that multiple workflows can point to.

## Use References For

- Browser QA tactics.
- Commit or PR wording principles.
- Tool-specific caveats.
- Express Light Rail backend concepts.
- Cross-workflow conventions.

Keep task procedures in `../workflows/`, implementation plans in `../plans/`, and copyable examples in `../templates/`.

Delete stale references when the underlying model or local project docs no longer need the hint. References should stay durable and hard to rediscover, not become a second README.
